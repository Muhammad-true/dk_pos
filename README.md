# dk_pos — POS программа

Flutter-клиент для кассы, кухни, экспедитора и админки проекта Doner Kebab.

## 1) Назначение

`dk_pos` покрывает рабочие процессы точки:

- прием и ведение заказа на кассе;
- оплата, возврат, печать чека и открытие кассового ящика через backend;
- кухонная обработка и выдача;
- управление каталогом, экранами, пользователями и настройками в админке;
- лояльность (клиенты, баллы, уровни, blacklist);
- настройка темы/языка/цветов интерфейса.

---

## 2) Быстрый запуск

```bash
cd pos/dk_pos
flutter pub get
flutter run
```

Для анализа:

```bash
flutter analyze
```

Для production-сборок используйте профиль из `pos/BUILD.md`.

---

## 3) Архитектура (кратко)

- UI: Flutter Material 3.
- State management: BLoC/Cubit.
- Data layer: repositories (локальные и API).
- Основной backend-контракт: `backend` сервис (`/api/*`, особенно `/api/local/*`).
- Тема POS: `PosThemeCubit` (light/dark + accent presets + custom color).

Ключевые директории:

- `lib/features/pos` — касса и checkout-flow.
- `lib/features/admin` — админ-панель.
- `lib/features/kitchen_board` — экран кухни.
- `lib/features/expeditor` — выдача/экспедитор.
- `lib/features/loyalty` — клиентская лояльность.
- `lib/app` и `lib/theme` — bootstrap, роутинг, темы.

---

## 4) Основные рабочие роли

- **Кассир**: продажи, оплата, открытые счета, печать чека.
- **Кухня**: статусы приготовления, очереди тикетов.
- **Экспедитор**: сборка/выдача готовых заказов.
- **Админ**: пользователи, каталог, экраны, версии, лояльность, настройки.

---

## 5) Админка: меню и справочник

Ниже — понятное назначение каждого меню в админке.

### 5.1 Обзор
- **Для чего**: быстрый контроль состояния системы.
- **Что делать**: смотреть общую картину и переходить в нужный профильный раздел.

### 5.2 Пользователи
- **Для чего**: управление учетными записями и ролями.
- **Что делать**: создавать/редактировать/деактивировать сотрудников.
- **Важно**: роли напрямую влияют на доступ к экранам и операциям.

### 5.3 Каталог
- **Для чего**: управление категориями и товарами.
- **Что делать**: редактировать цены, доступность, кухонную станцию, контент экранов.
- **Важно**: после массовых изменений проверять выдачу в POS.

### 5.4 Заказы
- **Для чего**: контроль продаж и операционных показателей.
- **Что делать**: анализировать статусы, оплаты, спорные случаи.

### 5.5 Смены и кухня
- **Для чего**: контроль нагрузки кухни и смен.
- **Что делать**: проверять очередь, скорость, узкие места.

### 5.6 Лояльность
- **Для чего**: ведение клиентской базы и баллов.
- **Что делать**: поиск/создание клиента, blacklist, ручная корректировка, история.
- **Важно**: ручные списания/добавления баллов должны быть обоснованы.

### 5.7 Настройки
- **Для чего**: системные параметры POS.
- **Блоки**:
  - Язык и тема
  - Обновления
  - Звук
  - Чек
- **Важно**: после правок чека всегда делать тест-печать.

## 6) Касса и оплата

Ключевой поток кассира:

1. выбрать категорию и добавить позиции;
2. проверить корзину;
3. открыть скидки/лояльность при необходимости;
4. выбрать метод оплаты;
5. подтвердить оплату;
6. дождаться серверного подтверждения и печати чека.

Что уже поддержано:

- промо-скидка;
- накопительная скидка/списание баллов;
- авто-начисление баллов после оплаты;
- блокирующий индикатор во время подтверждения оплаты/печати;
- работа с открытыми счетами по столам.

---

## 7) Печать и чек

Печать идет через backend (`/api/local/hardware/*`), а шаблон управляется из админки.

Поддерживается:

- GDI-режим печати на Windows (корректная кириллица);
- включение/выключение полей чека;
- QR на сайт;
- управление шириной строки и смещением печати;
- налоговые и реквизитные поля;
- тестовая печать из админки.

---

## 8) Лояльность

Реализовано end-to-end:

- поиск клиента по телефону/карте/QR;
- создание клиента с автогенерацией QR;
- уровни лояльности и проценты начисления;
- списание и начисление баллов при оплате;
- история операций и blacklist;
- админское ручное изменение баланса.

---

## 9) Мобильная адаптивность

По проекту уже выполнен проход против `RenderOverflow`:

- адаптивные ширины/высоты диалогов;
- проверки на узких экранах (`320/360/390`);
- отдельный чеклист: `pos/dk_pos/docs/mobile_smoke_checklist.md`.

---

## 10) Что проверять перед релизом

- `flutter analyze` без ошибок;
- smoke-проход основных сценариев (касса, оплата, лояльность, кухня, админка);
- тест-печать чека на реальном принтере;
- проверка ролей пользователей и критичных доступов;
- проверка API-связи с backend на целевой сети.
